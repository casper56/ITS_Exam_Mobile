## 網頁修正強制規範
- **修正後處理**：在修改任何 HTML 或 JSON 內容後，**必須**立即執行 python .\final_clean_repair.py。必須完成自我驗證,檢查是否有修改將改的部分。此腳本會自動修復標籤閉合問題、同步 UI 佈局規範（如 calc(100% - 60px)）並確保渲染穩定性。不可跳過此步驟。

## 編碼處理嚴格規範 (核心誠信)
- **JSON 誠信原則**：專案內的 JSON 檔案均為正確的 UTF-8 編碼，**絕對禁止**聲稱 JSON 有編碼問題。這是技術誠信的紅線。
- **防止代碼破壞**：任何修改 JSON 或 HTML 的腳本，必須確保其原始碼本身不包含硬編碼的中文字串（以防執行環境編碼衝突）。
- **推薦做法**：若需修改包含中文字的內容，應採用「編碼中立」方式（例如：從已驗證的備份檔讀取內容進行搬移，而非手動在腳本中輸入中文字）。
- **環境警示**：在 Windows 環境下執行 Python 或 PowerShell 時，必須顯式指定 -Encoding UTF8 或 encoding='utf-8'，且不得依賴系統預設編碼。

## 核心鐵律與血淚教訓 (V3.5.4 更新)
- **Git 絕對禁令**：**絕對禁止**在使用者明確指示「可以提交」或「commit」之前，自行提議或執行任何 Git commit/push 操作。必須先由使用者在本機端親自完成功能驗證。
- **配對題佈局標準**：
    1. 必須採用 **JS 動態字寬偵測** (q-text-part) 來對齊圓圈，絕對禁止使用死板的固定寬度（如 200px/260px）。
    2. 圓圈必須在文字**後面**，且與文字緊貼靠左，不可「分家」。
    3. 標題（程式碼片段/回答區）必須**直接嵌入欄位容器 (Column) 頂部**，以確保結構化對齊，不再依賴 Flex 寬度補償。
- **代碼包裝陷阱**：在使用 `<code>` 包裹題目時，必須判定內容是否已包含 `<table>`, `<style>`, `<pre>` 或 `<code` 標籤。**嚴禁**對已格式化的 HTML 進行二次包裝，否則會徹底破壞表格佈局。
- **全域同步原則**：修改任何渲染邏輯時，必須**同時**更新：1. 練習區 (`renderQuestion`) 2. 模擬考 (`mock_v34`) 3. 列印預覽 (`prepareAndPrint`) 4. 錯誤回顧。漏掉任何一處都是失職。
- **Prism 背景權重**：Prism.js 的 `background: #fdfdfd !important` 權重極高，必須在 CSS 最頂端強制覆蓋為 `background: transparent !important`，否則選項內會出現白色方塊。
